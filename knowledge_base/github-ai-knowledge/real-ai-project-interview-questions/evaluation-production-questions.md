# AI 评测与生产化真实面试题

## Q1: AI 项目上线前怎么评测？

### 真实问法

“你怎么证明这个 AI 功能可以上线？只靠人工看几个 case 吗？”

### 面试官想听什么

- 离线 eval set：覆盖高频、长尾、边界、失败、安全 case。
- 自动指标：准确率、召回率、格式合规、groundedness、拒答率。
- 人工评审：领域专家标注和 pairwise preference。
- 回归测试：每次 prompt / 模型 / 检索变更都跑。
- 线上灰度：小流量、监控、回滚。

### 可继续追问

- eval set 从哪里来？
- LLM-as-judge 可靠吗？
- 人工标注成本高怎么办？

### 高质量回答要点

成熟项目会同时做离线和在线评测。离线用 golden set 做版本准入，线上用真实用户反馈、人工抽检、关键指标监控验证效果。LLM-as-judge 可用于初筛，但高风险业务必须有人审或规则校验。

## Q2: 如何设计 RAG / Agent 的线上观测体系？

### 真实问法

“线上用户反馈 AI 答错了，你怎么定位？日志里会记录哪些东西？”

### 面试官想听什么

- request_id / trace_id 串起全链路。
- 记录 query rewrite、retrieval hits、scores、rerank、prompt hash、model、latency、cost。
- Agent 记录每步 state、tool call、tool result、错误和重试。
- 输出记录安全检查、引用、置信度和用户反馈。
- 敏感数据脱敏和采样策略。

### 可继续追问

- prompt 全量记录会不会泄漏数据？
- 怎么把失败 case 自动进入评测集？
- 如何看出某个版本上线后质量下降？

### 高质量回答要点

AI 可观测性要能回答“为什么答成这样”。对 RAG 要看到召回了什么；对 Agent 要看到每一步做了什么；对 LLM 要知道用的 prompt、模型版本和输出校验结果。日志不能只存最终答案。

## Q3: 如何控制 LLM 成本？

### 真实问法

“你们日调用量上来后 token 成本爆了，怎么降成本而不明显降质量？”

### 面试官想听什么

- prompt 压缩，减少无效上下文。
- 模型分层，小模型处理简单任务。
- semantic cache / exact cache。
- 批处理和异步任务。
- 限制 max_tokens、top_k、rerank 候选。
- 监控每个功能、用户、租户成本。

### 可继续追问

- 缓存如何避免返回过期答案？
- 怎么判断一个 query 可以用小模型？
- 成本优化会不会降低准确率？

### 高质量回答要点

先做成本归因，找出 token 大户。通常最有效的是减少输入上下文、压缩历史、控制 RAG chunk 数量和模型分层。缓存要绑定知识库版本、用户权限和模型版本，避免省钱但答错。

## Q4: AI 服务延迟怎么拆解和优化？

### 真实问法

“用户觉得 AI 很慢。你怎么分析是前端、检索、rerank、模型还是工具调用慢？”

### 面试官想听什么

- trace 分段：网关、检索、embedding、rerank、tool、LLM、后处理。
- P50/P95/P99 分开看。
- 流式输出降低首 token 延迟。
- 并发和缓存优化检索。
- 异步处理长任务。

### 可继续追问

- 首 token 慢和总耗时慢分别怎么优化？
- 工具调用不可控时怎么办？
- P99 尾延迟怎么治理？

### 高质量回答要点

先埋点再优化。LLM 应用常见问题是首 token 慢、上下文过长、rerank 候选过多、工具串行调用。可以通过 streaming、并发召回、减少上下文、模型路由和超时降级优化体验。

## Q5: AI 功能如何灰度和回滚？

### 真实问法

“你改了 prompt、换了模型或改了检索策略，怎么上线？出问题怎么回滚？”

### 面试官想听什么

- prompt/model/retriever 都有版本号。
- 小流量灰度和 A/B test。
- 指标阈值自动报警。
- 配置化开关和快速回滚。
- 保留旧索引、旧 prompt、旧模型路由。

### 可继续追问

- 怎么比较两个版本的质量？
- 用户反馈和自动指标冲突怎么办？
- RAG 索引版本如何回滚？

### 高质量回答要点

AI 变更必须版本化。上线前跑离线 eval，通过后按用户、租户或流量灰度。监控拒答率、错误率、用户负反馈、成本和延迟。RAG 索引用 alias 或版本号，出问题切回旧版本。

## Q6: 如何做 AI 项目的事故复盘？

### 真实问法

“线上出现了一次严重幻觉或错误工具调用，你怎么复盘并防止再次发生？”

### 面试官想听什么

- 保存完整 trace 和影响范围。
- 定位 failure mode。
- 添加 regression case。
- 修复 prompt、检索、工具权限或 guardrail。
- 增加监控和报警。

### 可继续追问

- 怎么判断是模型问题还是系统问题？
- 事故后是否需要人工审核一段时间？
- 如何评估修复有效？

### 高质量回答要点

不要把事故简单归因于“模型幻觉”。复盘要看输入、检索、上下文、prompt、模型输出、工具调用和后处理。修复后必须把 case 固化到 eval set，避免未来版本再次引入。
