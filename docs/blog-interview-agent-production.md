# 从本地 Demo 到生产级 AI 面试 Agent：Interview Agent 项目复盘

过去一段时间，我从一个很朴素的想法开始做了一个 AI 面试 Agent：它不只是简单地问几道八股题，而是能够读取候选人的简历、结合 AI 工程知识库、围绕真实项目持续追问，并根据回答质量判断要继续深挖、换方向，还是给出阶段性结论。

这个项目最后演进成了一个完整的工程化系统：后端有 AgentLoop、LangChain Harness、RAG、Guardrails、账号体系、积分计费、支付回调、生产存储；前端有 Electron + React 桌面端；底层有 PostgreSQL、Qdrant、MinIO、本地 EmbeddingService；同时也预留了微信小程序、iOS、Android、鸿蒙等多端方向。

这篇文章不是讲一个“Prompt 小玩具”，而是复盘如何把一个 AI Agent 做成一个更接近真实线上产品的工程。

## 一、为什么要做一个面试 Agent

很多 AI 面试工具的问题是：它们像题库，不像面试官。

真实面试里，面试官不会只问“什么是 RAG”“什么是向量数据库”。更关键的是：

- 你是否真的做过这个项目？
- 你在项目里的职责是什么？
- 为什么这么设计，而不是另一种方案？
- 线上指标是什么？
- 出过什么问题，怎么定位和复盘？
- 成本、延迟、召回率、权限、安全、灰度、监控有没有考虑？

所以这个项目的核心目标是：让 Agent 能基于简历和项目经历进行追问，而不是只进行泛泛问答。

最终我把产品定位成两种模式：

- `Agent 面试我`：Agent 作为面试官，基于简历和知识库对我追问。
- `Agent 回答我`：用户作为面试官提问，Agent 作为候选人，结合当前简历回答问题。

默认问答语言是中文，行业当前先聚焦互联网行业，知识重点放在 AI 应用工程、RAG、Agent、LLMOps、工程上线和安全治理。

## 二、整体架构

项目整体架构可以拆成几层：

```text
桌面端 / 小程序 / App
        |
        v
FastAPI Backend
        |
        +-- Account / Billing / Payment
        +-- Resume / Session / Memory
        +-- AgentLoop
              |
              +-- Harness Guardrails
              +-- LangChain Harness
              +-- RAG KnowledgeBase
              +-- Web Search
              +-- Model Runtime
        |
        +-- PostgreSQL
        +-- Qdrant
        +-- MinIO / S3
        +-- EmbeddingService
```

各模块的职责大概是：

| 模块 | 职责 |
| --- | --- |
| AgentLoop | 控制面试状态、阶段推进、追问和结束逻辑 |
| LangChain Harness | 封装模型调用、Prompt、工具上下文和降级逻辑 |
| Guardrails | 输入输出护栏，避免泄露评分标准、密钥和不合规内容 |
| RAG | 从 AI 面试知识库、历史记忆中检索上下文 |
| ResumeService | 解析和保存 PDF / Markdown 简历 |
| InterviewPersistenceService | 保存会话、轮次、历史 memory |
| BillingService | 账号、试用次数、积分、token 消耗和支付入账 |
| Desktop UI | Electron + React 对话界面 |

这个拆分的好处是：Agent 能力、业务数据、计费体系、存储基础设施是解耦的。后续要换模型、换向量库、换支付平台、扩展移动端，都不需要把核心逻辑推倒重来。

## 三、AgentLoop：让 Agent 像面试官一样工作

项目里最重要的一层是 `AgentLoop`。

如果只是把用户输入丢给大模型，模型很容易变成一个“聊天助手”。但面试官需要有明确的状态：

- 当前是开场、项目深挖、系统设计、行为面、总结，还是结束？
- 候选人刚才的回答是否足够具体？
- 是否需要继续追问当前项目？
- 是否应该切换到下一个考察方向？
- 是否需要给出阶段性评价？

因此我没有把所有逻辑都交给 LLM，而是显式维护面试状态：

```text
start()
  -> 生成第一题

step(candidate_answer)
  -> 判断是否是澄清问题
  -> 判断回答质量
  -> 生成追问 / 换题 / 总结
  -> 更新 state
  -> 持久化 turn
```

回答质量会关注一些生产化信号：

- 是否说明本人职责
- 是否有技术细节
- 是否有量化指标
- 是否有架构取舍
- 是否提到上线、灰度、监控、回滚
- 是否覆盖安全和边界问题

例如候选人只说“我做过 RAG 系统”，Agent 不应该马上换题，而应该追问：

- 文档是怎么切分的？
- embedding 模型怎么选？
- 召回和 rerank 怎么做？
- 如何评估召回率和回答忠实度？
- 线上延迟和成本怎么控制？
- 权限过滤在哪里做？

这也是 AgentLoop 的意义：它让 LLM 不只是生成一句话，而是在一个可控的面试流程里工作。

## 四、Harness：把模型调用变成可治理能力

项目中使用了 Harness 层来承接模型调用。

简单理解，AgentLoop 决定“下一步要做什么”，Harness 负责“怎么调用模型来完成这一步”。

Harness 里做了几件事：

- 组织系统 Prompt 和候选人上下文。
- 注入简历、项目经历、行业重点、RAG 检索结果。
- 调用不同模型供应商。
- 如果模型不可用，降级到 Scripted Harness。
- 输出后经过 Guardrails 检查。

我保留了两个 Harness：

- `LangChainInterviewHarness`：真实模型调用。
- `ScriptedInterviewHarness`：离线可测试，不依赖 API key。

这个设计非常实用。因为开发 Agent 时，很多逻辑其实不应该依赖真实模型才能测试。比如状态推进、会话持久化、账号扣费、简历隔离、RAG 检索，这些都可以在离线模式下验证。

## 五、RAG：知识库不应该手动触发

一开始很多人会把 RAG 理解成一个按钮：用户点一下“检索知识库”，再让模型回答。

但在这个项目里，RAG 是 Agent 的默认能力，不应该让用户感知成一个额外动作。

正常流程是：

```text
用户输入 / 当前面试问题
        |
        v
query expansion
        |
        v
BM25 recall + vector recall
        |
        v
hybrid score + MMR 去重
        |
        v
Top-K context 注入 Prompt
```

知识库来源主要包括：

- 拉取的 AI 面试指南文档。
- GitHub 上补充的 AI / RAG / Agent / LLMOps 资料。
- 用户历史会话中筛选出的有效问答 memory。

这里有一个重要取舍：历史对话不是全量记忆。

如果用户只是说“好的”“继续”“这个是什么”，这些内容没有长期价值，直接进入 memory 只会污染知识库。因此项目中增加了 `memory_decision`，只保留有实际问答内容、包含经验和信息密度的历史记录。

这样历史 memory 才能真正变成“下一次面试可检索的经验”，而不是聊天垃圾堆。

## 六、EmbeddingService 和 Qdrant

RAG 的核心能力离不开向量化。

在本地开发阶段，embedding 可以直接在 Python 进程里跑。但如果按生产思路设计，我更倾向于把 embedding 抽象成单独服务：

```text
Backend
  -> EmbeddingService
  -> Qdrant
```

这样做有几个好处：

- embedding 模型加载不阻塞主 API。
- 可以独立扩缩容。
- 可以单独做缓存和批处理。
- 便于将来替换成本地模型、OpenAI embedding 或云厂商 embedding。

向量数据库使用 Qdrant，主要存储知识库 chunk 和后续 memory 向量。PostgreSQL 存业务主数据，Qdrant 存向量索引，二者职责清晰：

- PostgreSQL：用户、简历、会话、订单、账本、知识元数据。
- Qdrant：向量检索。
- MinIO/S3：上传原文件。

这比把所有东西都塞进本地 JSON 文件更接近线上形态。

## 七、桌面端：从命令行到真实对话 UI

项目最初可以通过终端对话，但真正给用户使用，终端体验不够自然。

后来我做了 Electron + React 桌面端，核心界面包括：

- 左侧配置区：模式、行业、模型、简历、历史会话。
- 中间对话区：正常聊天式交互。
- 账号页：登录、试用次数、积分余额、模型计费说明。
- 简历库：上传多份简历，选择当前使用哪一份。

这里有一个产品设计点：充值、个人信息、登录状态不应该全部堆在主对话页。

所以我把账号相关能力单独放到 Account 页面，主页面只保留一个入口。这更接近真实产品，而不是一个工程 Demo。

桌面端当前的“开发充值”只用于本地调试，生产环境不会让客户端直接加余额，而是走支付订单和签名回调。

## 八、账号、试用和积分计费

如果一个 AI 产品准备上线，模型调用成本一定要治理。

这个项目里实现了基础的账号和计费体系：

- 用户可以邮箱注册和登录。
- 新用户默认获得 2 次试用。
- 试用用完后按模型 token 消耗扣积分。
- 每次模型生成都会记录 usage。
- 充值和消耗都写账本流水。

核心数据表包括：

```text
user_accounts
  用户账号、邮箱、密码 hash、试用次数、积分余额

recharge_orders
  充值订单，支持 pending / paid 状态

credit_ledger
  积分流水，充值为正，消耗为负

usage_records
  每次模型生成的 token、模型、成本、是否使用试用
```

计费单位使用 micros，避免小数精度问题：

```text
1 积分 = 1_000_000 micros
```

扣费时会优先使用模型供应商返回的真实 usage。如果没有 usage，就使用本地 token 估算器兜底。

## 九、支付为什么不能直接让客户端充值

这是项目生产化过程中非常关键的一步。

一开始为了本地调试，前端可以直接调用：

```text
POST /account/recharge
```

但这绝对不能作为生产充值方式。因为客户端传多少钱，后端就加多少钱，这等于把余额修改权限交给用户。

生产流程应该是：

```text
客户端创建支付订单
  -> POST /payments/orders
  -> 得到 external_order_id
  -> 调微信/支付宝/Stripe/Apple IAP
  -> 支付平台服务端回调
  -> POST /payments/webhook
  -> 后端验签
  -> 订单置为 paid
  -> 写 credit_ledger
  -> 增加用户余额
```

现在项目里已经把这条链路补齐：

- `/payments/orders` 创建 pending 订单。
- `/payments/webhook` 校验 `X-Payment-Signature`。
- `recharge_orders(tenant_id, external_order_id)` 做订单幂等。
- 重复支付回调不会重复加余额。
- `/account/recharge` 仅保留开发或管理员人工入账。

这才是一个可以继续接真实支付平台的后端边界。

## 十、安全和边界问题

准备上线时，安全不是一个“最后再说”的事情。

这次项目补了不少基础安全处理：

### 1. 生产环境启动自检

当设置：

```text
INTERVIEW_ENV=production
```

应用启动前会检查：

- 必须开启 API 鉴权。
- 不能使用默认 token secret。
- 禁止 dev-login。
- 禁止 mock provider login。
- 禁止 mock recharge。
- 必须配置支付回调密钥。
- 生产不能使用 SQLite。
- 生产不能使用 local object storage。
- CORS 不能是 `*`。

如果不满足，直接拒绝启动。

这个设计非常重要。很多线上事故不是代码不会写，而是危险配置被带到了生产环境。

### 2. 用户级数据隔离

数据查询统一按：

```text
tenant_id + user_id
```

隔离范围包括：

- 简历
- 面试会话
- 历史 memory
- 账号余额

同一个租户下，不同用户也不能互相读取简历和会话。

另外，创建面试时如果传了 `resume_id`，后端也会校验这份简历是否属于当前用户，避免引用别人上传的简历。

### 3. 扣费幂等

模型生成接口可能会被客户端、网关或网络重试。

如果没有幂等，用户同一次请求可能被扣两次积分。

现在项目会基于：

```text
X-Request-ID + session_id + event_type + turn_count + prompt + response
```

生成 `idempotency_key`，并在数据库里加唯一约束：

```text
usage_records(tenant_id, idempotency_key)
```

这样同一次生成重试不会重复消耗试用次数或积分。

### 4. 余额变动加锁

积分余额是钱相关数据，不能只靠应用层判断。

余额变动路径会锁住用户账户行，避免两个并发请求同时基于旧余额扣费或入账。

## 十一、多端方案思考

项目里除了桌面端，也预留了微信小程序、iOS、Android、鸿蒙方向。

如果真要上线，我会这样规划：

- 桌面端：Electron + React，适合重度练习和本地工具体验。
- 微信小程序：适合轻量获客和分享传播。
- iOS：SwiftUI 原生，接 Apple 登录和 IAP。
- Android：Kotlin + Jetpack Compose，接 Google Play / 国内支付。
- 鸿蒙：ArkTS，面向国内生态补充。

为什么不一定用 React Native？

因为这个产品不是一个简单壳应用。它涉及登录、文件上传、支付、系统权限、平台审核、原生体验、可能还有本地文件选择和推送。对准备认真上线的产品来说，关键平台用原生实现会更稳，长期维护成本也更可控。

当然，后台 API、账号体系、计费体系、RAG 和 AgentLoop 都是共用的。多端只是不同入口，不应该复制业务逻辑。

## 十二、项目里最大的收获

做完这个项目后，我对 AI Agent 工程有几个更明确的判断。

第一，Agent 不是一个 Prompt。

Prompt 很重要，但真正让系统可控的是状态机、工具边界、检索质量、失败降级、评测和持久化。

第二，RAG 不是“加一个向量库”。

真正的 RAG 要考虑知识来源、chunk、召回、重排、权限、更新、评测、引用和 memory 污染问题。

第三，生产级 AI 应用必须有成本系统。

只要调用大模型，就一定要面对 token 统计、模型价格、余额扣费、试用策略、异常消费和对账。

第四，安全不是上线前补一个鉴权。

简历是敏感数据，支付是资金数据，模型输出也可能带来风险。登录、数据隔离、支付验签、幂等、限流、审计、CORS、环境配置都要一起考虑。

第五，好的 AI 产品要既像产品，也像工程。

用户看到的是一个清爽的对话 UI，但背后需要数据库、对象存储、向量库、任务流程、计费账本、安全边界和可观测性支撑。

## 十三、后续计划

这个项目目前已经具备生产学习项目的基本形态，但离公开大规模上线还有一些事情要继续补：

- 接真实支付平台，例如微信支付、支付宝、Stripe、Apple IAP。
- 接正式短信验证码、微信登录、Apple 登录和账号绑定。
- 增加管理后台，用于用户管理、订单查询、人工调账和审计。
- 把模型价格目录从代码迁移到数据库或后台配置。
- 增加更完整的 API 访问日志、trace_id、异常告警。
- 把知识库自动更新做成后台任务。
- 把 memory 异步向量化到 Qdrant。
- 增加 RAG 命中日志和回答引用追踪。
- 做更系统的 LLM 输出评测和回归测试。

## 结语

Interview Agent 最开始只是一个“帮我面试”的想法，但真正做下去之后，它逐渐变成了一个完整的 AI 应用工程样板。

它里面有 AgentLoop，有 Harness，有 RAG，有 Guardrails，有桌面端，有账号，有支付，有计费，有数据隔离，也有生产环境自检。

这也是我认为 AI 应用开发最有意思的地方：模型能力只是起点，真正决定产品能不能上线、能不能长期运行的，是那些看起来不那么炫但非常扎实的工程系统。

如果把大模型看成发动机，那么一个可上线的 AI Agent 产品，还需要方向盘、刹车、仪表盘、燃油系统、维修手册和安全带。

这个项目，就是一次把“发动机”装进一辆真正能开上路的车里的过程。
