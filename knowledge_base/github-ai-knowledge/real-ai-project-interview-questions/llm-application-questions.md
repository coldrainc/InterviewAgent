# LLM 应用项目真实面试题

## Q1: 你如何设计一个面向生产的 LLM 应用架构？

### 真实问法

“如果让你做一个企业内部 AI 助手，从用户请求到模型返回，中间有哪些模块？每个模块解决什么问题？”

### 面试官想听什么

- API gateway、auth、rate limit、request validation。
- prompt orchestration、RAG、tool calling、model router。
- guardrails、safety filter、PII redaction。
- cache、retry、fallback、timeout。
- logging、trace、eval、feedback loop。
- async jobs、streaming、conversation store。

### 可继续追问

- 多模型路由怎么做？
- 什么时候用小模型，什么时候用大模型？
- 如何控制 token 成本？

### 高质量回答要点

生产 LLM 应用不是直接从前端调用模型。请求先经过鉴权、限流、输入治理，再进入 orchestration 层做 prompt、检索、工具调用和模型选择。模型输出还要经过格式校验、安全检查和引用校验，最后写入 trace 和反馈系统，用于评测与持续优化。

## Q2: Prompt 在项目中如何工程化管理？

### 真实问法

“你们 prompt 是写死在代码里，还是有版本管理？改 prompt 怎么评估不会引入回归？”

### 面试官想听什么

- prompt template versioning。
- 变量 schema 和输入校验。
- prompt registry 或配置中心。
- prompt 变更要跑 eval set。
- 线上灰度和 A/B test。

### 可继续追问

- prompt 和业务配置怎么区分？
- 谁有权限改 prompt？
- prompt 变更导致输出格式变了怎么办？

### 高质量回答要点

成熟方案会把 prompt 当成代码或配置资产管理：有版本、owner、review、eval、灰度和回滚。每次改 prompt 要跑固定评测集，检查正确率、格式合规率、拒答率、成本和延迟。对结构化输出，必须有 parser 和 schema validation。

## Q3: 结构化输出不稳定怎么解决？

### 真实问法

“你要求模型返回 JSON，但线上经常多一句解释、少字段、字段类型错。怎么保证稳定？”

### 面试官想听什么

- JSON schema / function calling / structured output。
- parser 校验和自动修复。
- 重试策略：带错误信息让模型修正。
- 默认值和兼容字段。
- 对关键流程不要完全信任模型输出。

### 可继续追问

- 如果重试 3 次还不合法怎么办？
- schema 升级如何兼容旧客户端？
- 字段内容合法但业务语义错怎么办？

### 高质量回答要点

优先使用模型或 SDK 支持的结构化输出能力，而不是纯 prompt 约束。输出后做 schema validation，失败时把错误摘要反馈给模型进行修复；仍失败则降级为人工处理或默认策略。业务关键字段还要做二次规则校验。

## Q4: 函数调用 / Tool Calling 怎么做可靠？

### 真实问法

“模型决定调用哪个函数，你怎么保证它不会漏调、错调、乱传参数？”

### 面试官想听什么

- tool schema 清晰，工具数量控制。
- 参数 enum、范围、必填字段。
- tool choice 可在关键流程强制指定。
- 调用前业务校验，调用后结果校验。
- trace 中记录每次 tool call。

### 可继续追问

- 一个问题需要多个工具时怎么编排？
- 工具描述写得越详细越好吗？
- 模型选择错工具怎么评估和修复？

### 高质量回答要点

不要把所有内部 API 都暴露给模型。工具应按任务场景裁剪，并提供清晰 schema。对关键路径，外层 workflow 可以决定必须调用哪些工具，模型只负责填参数或解释结果。工具调用失败要有 retry、fallback 和用户可理解的错误。

## Q5: 如何做多轮对话上下文管理？

### 真实问法

“用户多轮对话时，历史越来越长。你怎么决定哪些历史放进 prompt？怎么避免上下文漂移？”

### 面试官想听什么

- 短期窗口：最近 N 轮。
- 长期摘要：阶段性 conversation summary。
- 可检索记忆：向量化历史片段。
- 当前任务状态：结构化 state 优先于原始聊天记录。
- 明确重置和话题切换。

### 可继续追问

- 摘要错了怎么办？
- 历史里有用户纠正，怎么保证不再用旧信息？
- 多会话之间是否共享记忆？

### 高质量回答要点

生产系统不应把完整历史无脑塞进 prompt。更可靠的是维护结构化 state，比如用户目标、已确认事实、待办事项、约束条件；再配合最近对话窗口和可检索记忆。摘要和 memory 都要带时间、来源和置信度。

## Q6: 如何做模型 fallback 和多模型路由？

### 真实问法

“模型服务超时、限流、成本太高、质量波动时，你怎么保证系统可用？”

### 面试官想听什么

- 按任务类型路由：分类/抽取用小模型，复杂推理用大模型。
- 按 SLA 路由：低延迟任务优先快模型。
- 超时、限流、错误码触发 fallback。
- 不同模型 prompt 兼容性管理。
- 记录模型版本、输入输出和成本。

### 可继续追问

- fallback 到弱模型会不会质量下降？
- 怎么评估路由策略？
- 多模型输出不一致怎么办？

### 高质量回答要点

模型路由不是简单 try-catch。应基于任务复杂度、用户等级、成本预算、延迟 SLA 和历史质量选择模型。fallback 需要明确降级语义：是返回简化答案、排队异步处理、还是提示用户稍后再试。每种模型都要有独立 eval。
