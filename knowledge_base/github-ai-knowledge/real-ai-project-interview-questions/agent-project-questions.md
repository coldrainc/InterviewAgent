# Agent 项目真实面试题

检索关键词：真实 Agent 项目面试、AgentLoop、避免死循环、重复工具调用、终止条件、工具权限、Agent 记忆、多 Agent 协作、Agent 评测、ReAct 到工作流演进。

## Q1: 你做过的 Agent 项目和普通 Chatbot 有什么区别？

### 真实问法

“你简历上写了 Agent，不要只说用了 LangChain。这个 Agent 能自主完成什么任务？它的状态、工具、记忆、终止条件是怎么设计的？”

### 面试官想听什么

- Agent 不是聊天壳，而是有目标、状态、工具、计划、执行、观察、终止的系统。
- 能讲清楚任务边界：客服工单、代码修复、数据分析、面试官、运营助手、知识库维护。
- 有工具 schema、权限、重试、幂等、超时、审计。
- 有状态机或工作流，不是无限 ReAct 循环。
- 有可观测性：每步 tool call、输入输出、耗时、错误。

### 可继续追问

- 为什么用 Agent，而不是固定工作流？
- Agent 什么时候会失控？怎么限制？
- 工具调用失败会怎么恢复？
- 怎么判断任务完成？

### 高质量回答要点

强回答会把 Agent 设计成“可控的工作流 + 局部智能决策”。例如用状态机限定阶段：理解任务、检索资料、制定计划、调用工具、校验结果、生成回答。模型只在需要判断和生成时介入，关键副作用操作需要权限校验和人工确认。

## Q2: AgentLoop 怎么设计，才能避免死循环？

### 真实问法

“Agent 一直想、一直调用工具、一直修正，成本和延迟爆炸。你怎么设计 loop 的终止条件？”

### 面试官想听什么

- max_steps、max_tokens、max_wall_time、max_tool_errors。
- 状态收敛检测：连续 N 步没有新信息就停止。
- 工具调用去重：相同 tool + 相同参数不重复调用。
- 任务完成判定：结构化 checklist 或 verifier。
- 降级路径：返回部分结果、请求用户补充、人工接管。

### 可继续追问

- 怎么判断“没有新信息”？
- 如果模型反复调用同一个搜索工具怎么办？
- 你会把终止逻辑交给模型判断吗？

### 高质量回答要点

生产环境不能只靠模型说“我完成了”。应该在外层 AgentLoop 做硬约束：最大步数、超时、预算、重复工具检测；在内层用 verifier 判断目标是否满足。模型可以给 completion signal，但系统必须有 deterministic guardrails。

## Q3: 工具调用如何设计 schema 和权限？

### 真实问法

“你给 Agent 接了搜索、数据库、发消息、下单这些工具。怎么避免它乱调用或者传错参数？”

### 面试官想听什么

- 工具 schema 用强类型，参数有 enum、范围、必填、说明。
- 工具分级：只读、低风险写、高风险写。
- 高风险操作需要 human-in-the-loop。
- 参数校验、dry-run、幂等 key、审计日志。
- 工具结果要结构化，避免模型解析自然语言失败。

### 可继续追问

- 删除、转账、发通知这类操作怎么保护？
- 工具返回 500 或超时时怎么处理？
- 工具结果太长怎么给模型？

### 高质量回答要点

工具不是简单函数暴露，而是一个受控 API 层。模型只能看到最小必要工具；每个工具有 schema、权限策略、超时、重试和审计。写操作先 dry-run，展示计划和影响范围，用户确认后再执行，并使用幂等 key 防止重复提交。

## Q4: Agent 记忆怎么做，什么内容不应该进入 memory？

### 真实问法

“你说 Agent 有长期记忆。你保存什么？怎么检索？怎么避免把无意义寒暄和敏感信息也存进去？”

### 面试官想听什么

- 记忆类型：事实记忆、偏好记忆、任务记忆、经验复盘、失败案例。
- 写入前过滤：内容价值、稳定性、敏感性、可复用性。
- 检索时按当前任务 query、时间、用户、权限过滤。
- 支持遗忘、过期、用户删除。
- 记忆需要摘要，不是保存全部聊天噪音。

### 可继续追问

- 怎么判断一段对话值得记忆？
- memory 错了怎么办？
- memory 和 RAG 知识库有什么区别？

### 高质量回答要点

记忆写入应该是一个独立 decision step。只有包含可复用事实、用户偏好、明确决策、问题解决方案、失败复盘的内容才进入 memory；寒暄、短句、重复确认、敏感凭据、临时状态应丢弃。检索时 memory 只是上下文候选，不能覆盖当前用户明确输入。

## Q5: 多 Agent 协作项目怎么避免互相扯皮？

### 真实问法

“你用了多个 Agent：规划、检索、执行、审核。它们之间怎么协作？怎么避免大家互相重复工作或责任不清？”

### 面试官想听什么

- 明确角色边界和输入输出协议。
- central planner / supervisor 控制路由。
- 每个 Agent 输出结构化结果，不是自由聊天。
- 共享状态和任务队列，而不是彼此私聊。
- 有冲突解决和最终仲裁。

### 可继续追问

- 为什么不用一个大 Agent？
- 多 Agent 增加了哪些成本？
- 审核 Agent 真的能发现问题吗？怎么评估？

### 高质量回答要点

多 Agent 适合任务天然分工明确、需要并行或互审的场景；否则会引入额外复杂度。好的设计是 supervisor 分配任务，worker 输出结构化 artifact，reviewer 按 checklist 校验，最终由 supervisor 汇总。不要让 Agent 无限辩论。

## Q6: Agent 项目如何做评测？

### 真实问法

“普通问答可以看答案对不对。Agent 会调用工具、执行多步任务，你怎么评估它好不好？”

### 面试官想听什么

- 任务成功率，而不只是文本相似度。
- 工具调用正确率、参数正确率、步骤数、成本、耗时。
- 轨迹评估：是否调用了必要工具，是否调用了危险工具。
- 端到端评测 + step-level 评测。
- 回放真实失败 case，形成 regression suite。

### 可继续追问

- 如何构造 Agent eval dataset？
- 怎么评估工具参数是否正确？
- 线上如何发现 Agent 退化？

### 高质量回答要点

Agent eval 应该包含输入任务、初始环境、允许工具、期望最终状态、禁止行为和评分规则。除了最终回答，还要检查 tool trace：是否少调用、多调用、错调用、重复调用。生产中需要保存 trajectory，失败 case 自动进入回归集。

## Q7: 什么时候应该从 ReAct Agent 改成工作流？

### 真实问法

“很多项目一开始用 ReAct，后来发现不稳定。你怎么判断该不该换成 LangGraph / 状态机 / workflow？”

### 面试官想听什么

- 任务步骤固定、合规要求强、错误代价高时适合工作流。
- ReAct 适合开放探索、信息收集、不确定路径。
- 可以混合：外层 workflow，局部节点用 Agent 决策。
- 工作流更可测、可观测、可恢复。

### 可继续追问

- workflow 会不会损失灵活性？
- 状态持久化怎么做？
- 节点失败后怎么恢复？

### 高质量回答要点

成熟项目常见演进是：demo 阶段用 ReAct 快速验证；生产阶段沉淀成确定性工作流，把高风险路径固化，把模型决策限制在小范围节点。这样既保留智能性，也能提升稳定性和可运维性。
