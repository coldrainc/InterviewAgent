# AI 安全、治理与合规要求

## 风险管理框架

行业正在强调 AI 系统的治理闭环：

```text
Govern -> Map -> Measure -> Manage
```

也就是：

- 明确责任、角色、审批和风险等级。
- 识别系统用途、用户、数据、影响范围。
- 度量质量、安全、公平、鲁棒和隐私风险。
- 通过控制措施、监控、复盘持续管理。

## LLM 常见风险

### 1. Prompt Injection

攻击者通过用户输入、网页、文档、工具返回内容诱导模型忽略原规则或执行恶意操作。

防护：

- 把外部内容标记为 untrusted data。
- 不允许检索内容覆盖 system instruction。
- 工具调用前做权限和策略检查。
- 对工具输出做清洗。
- 红队测试。

### 2. Sensitive Information Disclosure

模型可能泄露：

- 系统提示。
- 内部文档。
- 用户隐私。
- API key。
- 训练数据片段。
- 无权限上下文。

防护：

- PII 检测和脱敏。
- 最小日志。
- 权限过滤。
- secrets scanning。
- 输出检查。
- 数据保留周期。

### 3. Excessive Agency

Agent 权限过大，可能误删数据、误发消息、误执行交易。

防护：

- 最小权限。
- 工具风险分级。
- human approval。
- 幂等和审计。
- 限制 step / cost / time。

### 4. Supply Chain

风险来自：

- 第三方模型。
- 开源依赖。
- prompt 模板。
- MCP server。
- embedding 模型。
- 数据源。

防护：

- 依赖版本锁定。
- 供应商评估。
- 模型和数据来源记录。
- server allowlist。
- SBOM / dependency scan。

## 数据治理要求

### 数据分类

至少区分：

- Public。
- Internal。
- Confidential。
- Restricted。
- Regulated。

### 数据流向

必须知道：

- 哪些数据进入 prompt。
- 哪些数据进入 embedding。
- 哪些数据写入日志。
- 哪些数据发给外部 provider。
- 哪些数据进入 memory。
- 哪些数据可被用户删除。

### 向量数据

embedding 向量也可能泄露语义信息，因此不能当作无敏感数据处理。

治理要求：

- 向量和原文同权限。
- 删除原文时删除向量。
- 权限变更更新 metadata。
- 备份加密。
- 访问审计。

## 合规关注点

真实项目需要关注：

- 用户是否知道 AI 参与。
- 高风险决策是否有人审。
- 是否能解释输出依据。
- 是否有申诉和纠错机制。
- 是否记录关键操作日志。
- 是否有数据删除和保留策略。

## 面试高频问题

### Q1: 只靠 prompt 能防 Prompt Injection 吗？

不能。prompt 是一层提示，不是安全边界。必须配合权限、工具策略、输入输出检查和审计。

### Q2: AI 系统为什么需要审计日志？

因为 AI 输出不可完全预测，审计日志用于复现问题、追责、合规证明和事故复盘。

### Q3: 如何处理高风险 AI 决策？

高风险场景需要人工审核、可解释依据、拒答机制、用户申诉、日志保留和严格评测。
